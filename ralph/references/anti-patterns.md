# 反模式与故障排查

## 常见反模式

| 反模式 | 后果 | 预防 |
|--------|------|------|
| 模糊 prompt（"做一个好的 API"） | 即刻失败或无限循环 | 用三要素公式：范围 + 证据 + 测试 |
| 过大范围（"重构整个系统"） | Agent 振荡，无法收敛 | 拆分为可独立验证的子任务 |
| 无 backpressure（无测试/lint） | Agent "凭感觉提交" | 每个条件必须对应一条验证命令 |
| 无卡住检测 | 在需人类决策处烧掉 50 次迭代 | 设置合理的 MAX_ITERATIONS |
| 复制别人的 prompt | 不适配你的项目 | 理解原理后自己写 |

## 迭代中故障信号

| 信号 | 诊断 | 处理 |
|------|------|------|
| 同一文件被修改 > 3 次 | 实现方案有误 | git checkout 该文件，换方案 |
| 测试反复在同一处失败 | 对代码理解有误 | 重新阅读相关代码和文档 |
| checklist 无进展 > 2 次迭代 | 任务拆分不合理 | 缩小每次迭代的目标 |
| 出现"过度烘焙"（添加无关功能） | 循环跑太久 | 收紧 prompt 约束 |
| state.json 被覆盖 | 两个循环写同一文件 | 每个任务用独立的 state 文件 |

## 诊断步骤

1. 查看 `.ralph-logs/` 中的迭代日志
2. 读取 `state.json` — 看 checklist 哪些通过了
3. 手动运行验证命令 — 确认验证链路本身没问题
4. 检查 PROMPT.md — 完成条件是否足够清晰

## 何时放弃 Ralph Loop

- 连续 3+ 次迭代无进展 → 切换到交互模式，人工诊断
- 验证命令本身有 bug → 先修复验证命令
- 需要架构决策 → 回到交互模式讨论
