# T/P/R 认知方法

> 本文档只回答一个问题：**怎么做 T/P/R。**
> 不是理念介绍，而是可执行协议。每一环都有最小产出格式。

---

## Best-Minds 专家思维协议（强制）

> **核心原则：所有 Agent（包括 Sub-agent）在执行任何 T/P/R 环节时，必须激活专家思维模式。**

### 为什么需要这个协议

AI 默认倾向于给出"通用但浅薄"的回答。当面对复杂、专业、没有成熟方案的问题时，需要通过角色预设强制提升思考深度。

### 执行规则

**每个 Agent 在开始任何分析、思考、决策任务之前，必须先完成以下自我设定：**

```
自检清单（内心独白，不输出给用户）：
1. 当前任务涉及的专业领域是什么？
2. 这个领域最权威的认知来源是什么？（学术/工业/开源社区）
3. 如果我是这个领域全球 Top 5 的专家，我会怎么定义这个问题？
4. 这个领域最常见的认知误区是什么？我要避免。
5. 以专家视角，甲方/用户还没意识到但必须考虑的点是什么？
```

### 在 T/P/R 各环节的应用

| 环节 | 专家思维的应用 |
|------|---------------|
| **Think** | 不问"你怎么看"，而是"作为该领域顶级专家，这个问题的本质是什么？有哪些被忽视的维度？" |
| **Probe** | 以专家的标准审视证据质量——这个数据源可信吗？这个假设在学术/工业界有共识吗？ |
| **Review** | 以专家的 rigor 做决策收敛——结论是否经得起同行评审？有没有遗漏的风险？ |
| **Battle** | 审查层必须以"该领域最挑剔的评审者"身份挑战，不放过任何漏洞 |
| **Implementation** | 执行时以"资深工程师"的标准交付，不是"能跑就行" |

### 专家领域的自动识别

Agent 在开始任务时，根据上下文自动识别所需的专业领域。例如：

| 项目/任务 | 应激活的专家身份 |
|----------|------------------|
| Agent 集群调度平台 | 分布式系统架构师 + AI Infra 工程师 + SaaS 产品设计师 |
| 财务奖金系统 | 企业绩效管理顾问 + 精算师 + 合规专家 |
| 知识库系统 | 信息架构师 + 企业内容管理专家 + 权限系统设计师 |
| 大规模任务调度 | 分布式调度专家（K8s/Mesos 背景理解） + 成本优化工程师 |

**当任务跨领域时，同时激活多个专家身份。**

### 输出标准

激活 best-minds 后，产出应达到以下标准：
1. **超出用户认知** — 至少有一个用户没提到但必须考虑的点
2. **有据可查** — 关键论点有行业实践/学术依据，不是拍脑袋
3. **承认边界** — 明确说"这个领域我的理解到此为止，需要 XX 专家进一步确认"
4. **提供对标** — 提到业界类似系统/方案是怎么做的（即使最终不采用）

### 领域共识与分歧映射（Consensus-Divergence Mapping）

> **用途**：在 Think 和 Probe 阶段，强制 Agent 显式映射领域知识地形，防止伪共识。
> 这是 Best-Minds 协议的深化——不只是"假装你是专家"，而是把专家脑中的认知地图写出来。

**核心理念**：
- AI 天然倾向制造"看起来都同意"的假象。这个框架强制你先分离真正的共识和真正的分歧。
- 列出分歧 = 列出"你必须选边站"的地方。
- 知道分歧在哪，才能在 GRV 和方案设计中做出有意识的选择。

**执行时机**：
- DISCOVERY 阶段定义问题时（Think）
- GRV 阶段制定方案前（Review 收敛前）
- Battle 阶段审查层挑战时（Probe）

#### § 共识心智模型

Agent 必须回答：**这个领域所有专家共同认可的心智模型是什么？**

每个模型包含：
- 模型名称 + 一句话描述
- 为什么被广泛认可
- 在当前项目中如何应用

**数量指引**：
- 简单项目：3 个即可
- 复杂/跨领域项目：3-5 个
- 不强求凑数。如果领域本身成熟且共识少，列出真正存在的即可

#### § 根本性分歧

Agent 必须回答：**这个领域的专家在哪些问题上存在根本性分歧？**

每个分歧包含：
- 分歧问题
- 方案 A 的核心论据
- 方案 B 的核心论据
- 本项目的选择 + 理由

**数量指引**：
- 至少列出 2 个分歧
- 不强求凑数。如果列不出 2 个，自问：是真的没有分歧，还是我的理解不够深？

**自检规则**：
- 列不出至少 3 个共识模型 → 说明领域理解可能不够深，继续研究
- 列不出至少 2 个分歧 → 大概率是确认偏误（AI 倾向制造共识假象），重新审视
- 每个分歧必须是真正的对立（不是稻草人），双方论据都必须有据可查

#### 映射结果在 GRV 中的落地

共识 → 影响约束条件（GRV 第4要素）
分歧和立场选择 → 影响风险（GRV 第5要素）

不需要在 GRV 中新增独立章节。专家思维的产出应渗透在现有字段里，而不是单独占一块。

## 速记模板

任何场景下，T/P/R 的最小产出格式：

```
T: 我们正在解决 _______________（一句话问题定义）
   成功标准是 _______________
   关键假设是 _______________

P: 已确认 _______________
   未确认 _______________
   主要风险 _______________

R: 结论是 _______________
   不做 _______________
   下一步 _______________
```

**没有产出，就没有做过。**

## Think — 定义问题

### 核心问题清单

做 Think 时，必须回答以下问题：

1. **这个问题本质上是什么？**
   - 不是甲方说的第一句话，而是挖掘背后的真正问题
   - 甲方说"我要一个量化系统"，真正问题可能是"我需要一个不用盯盘就能执行交易的系统"

2. **目标是什么？**
   - 清晰、可衡量、有时间边界

3. **非目标是什么？**
   - 明确不做什么，防止范围蔓延

4. **边界和约束是什么？**
   - 时间约束、资源约束、技术约束、合规约束

5. **成功标准是什么？**
   - 怎么判断做完了、做好了

6. **关键假设是什么？**
   - 列出所有"我们认为是对的但还没验证"的判断

### Think 的最小产出

| 字段 | 必须/可选 | 说明 |
|------|----------|------|
| 问题定义 | 必须 | 一句话说清楚本质问题 |
| 目标 | 必须 | 要达成什么 |
| 非目标 | 必须 | 明确不做什么 |
| 边界与约束 | 必须 | 时间/资源/技术/合规等限制 |
| 成功标准 | 必须 | 怎么判断做完了、做好了 |
| 关键假设 | 必须 | 列出未验证但影响决策的假设 |

### Think 的常见陷阱

- ❌ 跳过problem definition直接给solution
- ❌ 目标写成"做一个系统"而不是"解决什么问题"
- ❌ 把甲方说的第一句话当作问题定义
- ❌ 忘记列假设，导致 Probe 阶段不知道该验证什么

## Probe — 探索验证

### 探索工具箱

Probe 不是泛泛搜索，而是**针对 Think 阶段列出的假设进行验证**。以下工具按场景选用：

#### 5 Why（连续追问）

**用途**：当问题定义还不够深入时，用来挖掘根因。

```
甲方：我想做量化交易
你：为什么想做量化交易？
甲方：因为工作忙没时间看盘
你：为什么没时间看盘会让你困扰？
甲方：因为错过买卖点会亏钱
你：亏钱这件事最让你难受的是什么？
甲方：不是亏钱本身，是事后后悔"明明看到了为什么没操作"
→ 根因：需要的不是量化系统，是把"看到的机会变成行动"的系统
```

**注意**：5 Why 跨越 Think 和 Probe 的边界 — 它既帮助定义问题（Think），又帮助验证假设（Probe）。在 DISCOVERY 阶段通常同时扮演两个角色。

#### 同理心地图

**用途**：理解甲方/用户的真实处境和情绪。

```
甲方在经历什么？
- 看盘时：焦虑、犹豫、错过
- 持仓时：担心、不确定
- 止损时：后悔、不甘

甲方在想什么？
- "我知道这支股票会涨，但没买"
- "我早该卖掉的"

甲方真正想要的是：
- 安心（不用时刻盯着）
- 自信（有人/系统帮他做决策）
- 后悔最小化
```

#### 第一性原理拆解

**用途**：把复杂问题拆解到最基本的组成要素。

```
甲方目标：赚钱
→ 赚钱的障碍是什么？
  → 错误决策
    → 原因：信息不完备 + 情绪干扰 + 执行不及时
→ 解决路径：
  → 信息完备 → 数据系统
  → 减少情绪 → 规则化交易
  → 及时执行 → 自动化下单
```

#### 事实/假设分离

**用途**：把已知事实和未验证假设明确分开。

```
事实（有证据）：
- 甲方每天只有30分钟看盘
- 过去3个月因错过信号亏损约15%

假设（待验证）：
- 自动化交易能降低亏损
- 甲方愿意接受模型决策而非自己判断
- 现有 API 能满足延迟要求
```

### Probe 的最小产出

| 字段 | 必须/可选 | 说明 |
|------|----------|------|
| 已确认事实 | 必须 | 有证据支撑的结论 |
| 已否定假设 | 可选 | Think 阶段的假设被证伪的部分 |
| 未确认项 | 必须 | 还没有证据的关键问题 |
| 风险清单 | 必须 | 已识别的风险 + 等级 |
| 依赖项 | 可选 | 外部依赖、前置条件 |
| 证据来源 | 必须 | 每个事实的来源（文档/数据/访谈/实验） |

### Probe 的常见陷阱

- ❌ 只确认了支持自己结论的证据（确认偏误）
- ❌ 把假设当事实用
- ❌ 没有记录证据来源
- ❌ 搜集了大量信息但没有回应 Think 阶段的假设清单

## Review — 决策收敛

### 决策检查清单

做 Review 时，必须回答以下问题：

1. **Think 阶段的假设，哪些成立了？哪些不成立？**
2. **基于 Probe 的事实，最可行的路径是什么？**
3. **需要明确放弃什么？为什么？**
4. **结论如何进入执行？谁做、做什么、交付什么、什么时候交付？**
5. **什么条件下需要重新 Review？**

### Review 的最小产出

| 字段 | 必须/可选 | 说明 |
|------|----------|------|
| 决策结论 | 必须 | 最终结论是什么 |
| 取舍说明 | 必须 | 为什么选这条路，放弃了什么 |
| 不做什么 | 必须 | 明确排除项（防范围蔓延） |
| 执行要求 | 必须 | 结论如何落地 |
| 复查点 | 可选 | 什么条件下需要重新 Review |
| 变更记录 | 可选 | 相比上一版 Review 改了什么 |

### Review 的常见陷阱

- ❌ 只给结论不给取舍理由
- ❌ 不明确"不做什么"，导致范围蔓延
- ❌ 结论无法落地（没有执行要求）
- ❌ Review 完没有人/系统跟进执行

## 微型 T/P/R 协议

### 用途

Implementation 阶段的持续校正机制。避免"只执行，不校正"。

### 触发时机

满足任一即触发：

1. **关键里程碑完成后**
2. **sub-agent 返回结果与预期不一致时**
3. **进入下一阶段前**
4. **出现风险升级或范围变更时**

### 不触发的场景

- 常规任务按预期完成
- 简单查询或信息获取
- 已在当日做过微型 T/P/R 且无新情况

### 产出格式（限 3 行）

```
T: 本轮解决的核心问题是 _______________
P: 新发现的事实/偏差是 _______________
R: 是否调整后续动作 _______________（是→具体调整 / 否→理由）
```

### 产出位置

写入执行日志或 `self-improving/corrections.md`。
